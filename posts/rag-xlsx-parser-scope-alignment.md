# RAG용 Excel 파서를 만들 때 먼저 정해야 하는 것들

문서 파싱을 하다 보면 PDF나 PPTX보다 Excel이 더 단순해 보일 때가 있다. PDF는 레이아웃이 복잡하고, PPTX는 이미지와 도형이 많고, Word는 문단 구조가 제각각이다. 반면 Excel은 행과 열이 있으니 그냥 셀 값을 읽으면 될 것처럼 느껴진다.

그런데 실제 업무용 Excel을 RAG 파이프라인에 넣어보면 생각이 조금 달라진다.

Excel 파싱에서 어려운 점은 “파일을 열 수 있는가”가 아니다. 진짜 문제는 **어떤 Excel 기능까지 지원할 것인지 정하고, 그 범위를 사용자와 합의하는 것**이다.

## “Excel 파싱”이라는 요구사항은 너무 넓다

“Excel 파일을 파싱해 주세요”라는 말 안에는 생각보다 많은 기능이 들어 있다.

단순히 텍스트와 숫자 셀만 읽으면 되는 파일도 있다. 하지만 실제 업무 파일에는 다음 요소들이 섞인다.

- 여러 시트
- 텍스트, 숫자, 날짜
- 수식과 수식 결과값
- 병합 셀
- 다단 헤더
- 한 시트 안의 여러 테이블
- Excel Table 객체
- 이미지, 차트, 도형, 텍스트 박스
- 숨겨진 행/열/시트
- 하이퍼링크, 주석, 드롭다운
- 피벗 테이블
- 대용량 파일

이 요소들을 모두 같은 난이도로 보면 안 된다. “셀 값을 읽는 것”과 “차트를 해석하는 것”은 완전히 다른 일이다. 숨겨진 데이터를 포함할지 제외할지도 기술 문제라기보다 제품 정책에 가깝다.

그래서 Excel 파서를 만들 때 첫 번째로 해야 할 일은 특정 라이브러리를 고르는 것이 아니라, **어떤 Excel 기능까지 지원할지 범위를 나누는 것**이었다.

## 기능을 단계로 나누어야 한다

Excel 파싱 범위는 대략 이런 식으로 나눌 수 있다.

| 단계 | 범위 | 의미 |
|---|---|---|
| 1단계 | 기본 셀, 여러 시트, 기본 표 | 일반적인 텍스트/숫자 데이터 추출 |
| 2단계 | 수식 결과값 | 사용자가 화면에서 보는 값을 읽을 수 있는지 확인 |
| 3단계 | 병합 셀, 다단 헤더 | 표의 의미 구조를 보존하기 위한 핵심 영역 |
| 4단계 | 한 시트 안의 여러 테이블, Excel Table 객체 | 표 경계와 테이블 단위 분리 |
| 5단계 | 이미지, 차트, 도형, 텍스트 박스 | 값 추출보다 해석에 가까운 영역 |
| 6단계 | 숨김 데이터, 링크, 주석, 드롭다운, 대용량 파일 | 제품 정책과 운영 안정성 영역 |
| 7단계 | 실제 업무 파일 검증 | 스펙대로 구현했는지 확인하는 마지막 단계 |

이렇게 나누면 “Excel 파싱을 지원한다”는 모호한 요구를 “이번 버전에서는 어디까지 지원할 것인가”라는 선택 문제로 바꿀 수 있다.

## 클라이언트도 Excel 기능 범위를 처음부터 알기는 어렵다

개발자 입장에서는 Excel 기능을 하나씩 떠올릴 수 있지만, 클라이언트는 보통 “Excel 파일이 들어간다” 정도로 요구사항을 말한다. 그 말 안에 수식, 이미지, 차트, 숨김 시트, 피벗 테이블까지 들어가는지 처음부터 명확하지 않다.

그래서 클라이언트와 범위를 맞출 때는 기능별로 쪼개서 보여주는 편이 좋다.

예를 들어 다음처럼 설명할 수 있다.

- 기본 셀과 여러 시트는 비교적 빠르게 지원할 수 있다.
- 수식은 “수식 자체”를 볼 것인지, “화면에 보이는 결과값”을 볼 것인지 정해야 한다.
- 병합 셀과 다단 헤더는 표 구조 보존에 중요하므로 별도 처리가 필요하다.
- 이미지, 차트, 도형은 단순 추출이 아니라 해석에 가까우며 정확도 한계가 있다.
- 숨김 데이터나 주석은 기술적으로 읽을 수 있어도 제품에서 포함할지 정책을 정해야 한다.
- 대용량 파일은 파싱 품질뿐 아니라 처리 시간과 실패 복구까지 고려해야 한다.

이렇게 기능별로 나누고 각 항목의 예상 개발 시간을 보여주면, 클라이언트가 “필요한 기능”과 “나중에 해도 되는 기능”을 선택할 수 있다.

## 스코프 합의 없이 시작하면 요구사항이 계속 흔들린다

처음부터 “Excel 파싱 전체 지원”이라고 잡으면 나중에 질문이 계속 늘어난다.

- 차트도 읽을 수 있나요?
- 숨겨진 시트도 포함하나요?
- 피벗 테이블도 되나요?
- 이미지 안의 글자도 읽나요?
- 수식은 결과값을 읽나요, 수식 문자열을 읽나요?
- 한 시트 안에 표가 여러 개 있으면 어떻게 나누나요?

이 질문들은 구현 중간에 나오면 일정과 설계를 흔든다. 그래서 개발 전에 “Excel 파싱”을 기능 단위로 분해하고, 이번 범위에 포함할 것과 제외할 것을 명확히 해야 한다.

특히 RAG 파이프라인에서는 파서 출력이 검색 품질과 답변 품질로 이어진다. 따라서 단순히 많이 읽는 것보다, 이번 제품에서 필요한 구조를 안정적으로 읽는 것이 더 중요하다.

## RAG용 Excel 파서에서 확인해야 할 것

RAG에 넣을 Excel 파서를 설계할 때는 다음 항목을 따로 확인해야 한다.

### 1. 셀에 보이는 값을 읽을 것인가, 수식을 읽을 것인가

사용자는 보통 셀에 보이는 결과값을 기대한다. 하지만 파일 내부에는 수식과 캐시된 결과값이 섞여 있다. 외부 파일을 참조하는 수식이나 저장된 적 없는 계산 결과는 별도 리스크가 된다.

### 2. 표 경계를 어떻게 나눌 것인가

한 시트에 하나의 표만 있는 경우는 단순하다. 하지만 실제 Excel에는 한 시트 안에 여러 테이블이나 설명 영역이 섞여 있는 경우가 많다. 이때 어디까지가 하나의 표인지 판단해야 한다.

### 3. 병합 셀과 다단 헤더를 어떻게 표현할 것인가

사람이 보기에는 자연스러운 표도 파서 입장에서는 병합 셀, 빈 셀, 2단 헤더 문제가 된다. 이 구조를 잃으면 행과 열의 의미가 흐려질 수 있다.

### 4. 이미지와 차트를 어디까지 처리할 것인가

이미지나 차트는 셀 값처럼 추출하는 대상이 아니다. 필요하다면 별도 OCR이나 VLM 기반 설명이 들어갈 수 있지만, 이 경우 정확도와 비용을 함께 설명해야 한다.

### 5. 숨김 데이터와 메타데이터를 포함할 것인가

숨겨진 행/열/시트, 주석, 링크, 드롭다운은 “읽을 수 있는가”보다 “읽어야 하는가”가 먼저다. 사용자에게 보여지지 않는 데이터를 RAG에 포함하면 오히려 혼란을 만들 수 있다.

### 6. 실제 업무 파일로 검증할 것인가

샘플 파일에서 잘 되는 것과 실제 업무 파일에서 잘 되는 것은 다르다. Excel은 작성자 습관에 따라 구조가 크게 달라지기 때문에, 최종적으로는 실제 파일 기반 검증이 필요하다.

## 결국 파서 개발 전에 합의해야 하는 것

Excel 파서 개발에서 먼저 정해야 하는 것은 다음 세 가지다.

1. **이번 버전에서 지원할 Excel 기능 범위**  
   기본 셀만 볼 것인지, 수식/병합 셀/이미지/차트/숨김 데이터까지 볼 것인지 정한다.

2. **포함하지 않을 기능과 그 이유**  
   기술적으로 어려워서인지, 제품 가치가 낮아서인지, 정확도 리스크가 커서인지 명확히 한다.

3. **검증 기준**  
   어떤 실제 업무 파일에서, 어떤 결과가 나오면 성공으로 볼지 정한다.

이 세 가지가 정리되어야 이후에 어떤 라이브러리를 쓸지, 후처리를 어떻게 할지, LLM을 어디에 붙일지 판단할 수 있다.

## 결론

Excel 파싱은 단순히 `.xlsx` 파일에서 텍스트를 꺼내는 일이 아니다.

RAG 파이프라인에서 Excel 파서는 사람이 보던 표와 데이터를 검색 가능한 형태로 바꾸는 역할을 한다. 그래서 기술 구현 전에 먼저 “어떤 Excel 기능까지 지원할 것인가”를 정해야 한다.

기본 셀, 수식, 병합 셀, 여러 테이블, 이미지, 차트, 숨김 데이터는 각각 다른 난이도와 리스크를 가진다. 이 차이를 클라이언트와 공유하고, 기능별 개발 시간과 한계를 설명한 뒤 범위를 확정해야 한다.

그다음에야 어떤 파서를 쓸지, 어떤 후처리를 붙일지, 실제 파일 검증을 어떻게 할지 결정할 수 있다.

Excel 파싱을 시작할 때 가장 먼저 던져야 할 질문은 이것이다.

> 우리는 Excel을 “전부” 지원하려는 것이 아니라, 이번 제품에서 어떤 Excel 기능까지 책임지려는가?

이 질문에 먼저 답해야 “Excel 파싱 지원”이라는 넓은 요구사항을 실제 개발 가능한 제품 기능으로 만들 수 있다.
