# RAG용 XLSX 파서 비교: docling보다 openpyxl이 나았던 이유

## 이 글의 초점

Excel 파싱에는 두 가지 질문이 있다. 하나는 “어떤 Excel 기능까지 지원할 것인가”이고, 다른 하나는 “그 기능을 어떤 파서로 안정적으로 처리할 것인가”다.

이 글은 두 번째 질문에 집중한다. 즉, 실제로 `docling`과 `openpyxl`을 비교했을 때 왜 `openpyxl`이 더 나았는지 정리한다.

## 문제의식

XLSX를 RAG 파이프라인에 넣을 때 중요한 것은 단순히 “파일을 읽을 수 있는가”가 아니다.  
검색과 답변 품질을 위해서는 표의 의미 구조가 유지되어야 한다.

특히 업무용 Excel에는 다음과 같은 구조가 자주 나온다.

- 왼쪽 카테고리 열이 여러 행에 걸쳐 병합된 표
- 여러 시트에 나뉜 데이터
- 사람이 보기 좋게 만든 다단 헤더
- 빈 셀처럼 보이지만 병합 셀의 일부인 영역
- 표 경계가 명확하지 않은 여러 개의 데이터 블록

이런 파일에서는 파서가 “텍스트를 뽑았다”는 사실보다, **셀과 표의 관계를 얼마나 보존했는지**가 더 중요하다.

## 비교 결과 요약

테스트에서는 두 파서 모두 기본적인 실행은 통과했다. 하지만 결과물의 구조 품질과 처리 시간에서 차이가 났다.

| 비교 항목 | openpyxl | docling |
|---|---|---|
| 기본 실행 | 통과 | 통과 |
| 전체 처리 시간 | 더 짧음 | 더 김 |
| 평균 문서 처리 시간 | 더 빠름 | 더 느림 |
| 병합 셀 처리 | rowspan 보존 | rowspan 시맨틱 손실 |
| 테이블 수 | 원래 구조에 가까움 | 테이블이 과분리되는 경향 |
| 후처리 난이도 | LLM이 표 구조를 이해하기 쉬움 | LLM이 분리된 테이블을 재조합해야 함 |

여기서 핵심은 “openpyxl이 무조건 모든 상황에서 낫다”가 아니다.  
**테스트 대상처럼 병합 셀과 카테고리 열이 많은 업무용 XLSX에서는 openpyxl이 더 안정적이었다**는 점이다.

## 왜 openpyxl이 더 나았나

### 1. XLSX 내부 구조를 직접 다룬다

openpyxl은 XLSX 파일의 내부 XML 구조를 직접 읽는다. 그래서 `mergeCell` 같은 Excel 고유 정보를 활용할 수 있다.

병합 셀이 단순히 빈 칸으로 풀려버리면, 사람에게는 자연스럽던 표가 기계에게는 의미를 잃는다. 예를 들어 하나의 카테고리명이 여러 행에 걸쳐 적용되는 표에서, 병합 정보를 잃으면 각 행이 어떤 카테고리에 속하는지 불분명해진다.

### 2. rowspan을 보존할 수 있다

업무용 Excel에서는 왼쪽 열에 “기본 정보”, “상세 정보”, “주의사항” 같은 카테고리를 세로 병합해두고, 오른쪽 열에 세부 항목을 나열하는 형태가 많다.

openpyxl 기반 파싱은 이 병합 관계를 HTML의 `rowspan` 같은 구조로 보존할 수 있었다. 이러면 LLM 후처리 단계에서도 “이 행들이 같은 카테고리에 속한다”는 맥락을 유지할 수 있다.

### 3. docling은 테이블을 과분리하는 케이스가 있었다

docling은 여러 문서 포맷을 다루는 범용 문서 파서로는 장점이 있다. 하지만 테스트한 XLSX에서는 카테고리 열과 데이터 열을 별도 테이블처럼 분리하는 케이스가 있었다.

문제는 이렇게 분리된 뒤에는 LLM 후처리로도 원래 표의 의미를 완전히 복원하기 어렵다는 점이다.

- 카테고리만 있는 테이블
- 실제 데이터만 있는 테이블
- 둘 사이의 관계가 약해진 상태

이런 입력이 들어오면, 뒤에서 아무리 `merge`, `label`, `split` 같은 구조화 로직을 넣어도 원본 표의 의미가 흐려질 수 있다.

## LLM 후처리가 모든 것을 해결하지는 않는다

문서 파싱 파이프라인에서는 종종 “일단 텍스트를 뽑고 LLM으로 정리하면 되지 않을까?”라고 생각하기 쉽다. 하지만 XLSX에서는 원본 구조가 손실되면 LLM이 복원해야 할 정보가 너무 많아진다.

LLM은 구조를 정리할 수는 있지만, 파서가 이미 잘못 분리한 테이블의 관계를 항상 정확히 되살리지는 못한다. 그래서 RAG용 파서에서는 **LLM 이전 단계의 raw HTML 품질**이 중요하다.

## docling이 나쁜 선택이라는 뜻은 아니다

docling은 PDF, Word, PPTX 등 다양한 문서를 한 파이프라인에서 다룰 때 강점이 있다. 단순한 격자형 XLSX나 병합 셀이 거의 없는 파일에서는 충분히 쓸 수 있다.

다만 Excel 고유 구조가 답변 품질에 직접 영향을 주는 경우, 특히 병합 셀과 다단 헤더가 많은 업무용 XLSX에서는 openpyxl처럼 포맷 특화 라이브러리로 구조를 보존하는 접근이 더 유리할 수 있다.

## 결론

RAG용 XLSX 파서 선택 기준은 “파싱 성공 여부”가 아니라 “검색 가능한 의미 구조를 보존했는가”여야 한다.

이번 비교에서 openpyxl이 더 좋았던 이유는 단순히 빠르기 때문만이 아니라, 병합 셀과 표 구조를 더 잘 보존했기 때문이다. 반대로 docling은 범용 문서 파서로서 장점이 있지만, 테스트한 업무용 XLSX에서는 테이블 과분리와 컨텍스트 손실이 문제가 됐다.

따라서 XLSX 파서를 고를 때는 다음 질문을 먼저 던져야 한다.

> 이 파서는 Excel을 텍스트로 바꾸는가, 아니면 Excel의 표 구조를 RAG가 이해할 수 있는 형태로 보존하는가?
