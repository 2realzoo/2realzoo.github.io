# RAG를 위한 PDF 파서 선택 가이드

RAG(Retrieval-Augmented Generation) 파이프라인을 구축할 때 문서 파싱 품질은 검색 정확도에 직접적인 영향을 줍니다. 특히 **헤더 계층 구조 보존**과 **테이블 추출 정확도**가 RAG 청킹 전략의 핵심입니다.

이 글에서는 실제 문서 4종(디지털 PDF, 스캔 PDF, 한/영 혼합 논문)을 대상으로 세 가지 파서를 비교한 결과를 공유합니다.

---

## 테스트 환경

| 항목 | 내용 |
|------|------|
| 테스트 파서 | Docling 2.49 / Azure Document Intelligence (Layout S0) / opendataloader-pdf |
| 테스트 문서 | 문서A (14p, 디지털) / 문서B (44p, 디지털) / 문서C (1p, 스캔) / 논문D (5p, 한영혼합) |

---

## 1. 추출 문자 수 (완성도)

| 파일 | Docling | Azure DI | opendataloader |
|------|---------|----------|----------------|
| 문서A (14p, 디지털) | 25,779 | 25,086 | 24,109 |
| 문서B (44p, 디지털) | 66,969 | 58,982 | 77,941 |
| 문서C (1p, **스캔**) | **62** | **5,815** | **50** |
| 논문D (5p, **한/영 혼합**) | 17,555 | 17,125 | **445** |

주목할 점:
- **opendataloader 문서B 77,941자**: 숫자가 크지만 실질 정보량은 낮습니다. 병합 셀(colspan)을 여러 열로 쪼개면서 같은 내용이 3~4회 반복 삽입되고, 일반 텍스트 단락까지 테이블로 과잉 인식해 pipe 구분자(`|`, `---`)가 글자 수에 포함된 결과입니다.
- **Docling 문서C 62자**: 스캔 PDF에서 OCR 없이 모델 타임아웃 후 빈 결과 반환.
- **opendataloader 논문D 445자**: 텍스트 미인식, 이미지 경로 3개만 출력.

---

## 2. 헤더 계층 구조 ← RAG 청킹의 핵심

### Docling — 모든 헤더가 H2로 평탄화

```
## 섹션 1
## 섹션 2
## ○   ← 의미없는 기호도 헤더로 오인식
## 섹션 3
```

모든 헤더가 H2 단일 레벨로 나옵니다. `MarkdownHeaderTextSplitter` 같은 계층 청킹 전략을 쓸 수 없습니다.

### Azure DI — H1/H2/H3/H4 계층 완전 보존 ✅

```
# 주요 섹션
  ## 1. 소분류
  ## 2. 소분류
    ### 1) 세부항목
      #### (1) 하위항목
```

실제 문서 구조를 그대로 반영합니다. 청킹 전략을 계층별로 세밀하게 설계할 수 있습니다.

### opendataloader — 문서마다 편차가 큼

- **문서A**: 헤더 1개(H1)만 추출 → 전체 문서가 단일 덩어리, 청킹 불가
- **문서B**: H1~H6이 나오지만 레벨 순서가 불규칙 (H1→H6→H3→H2→H4)

---

## 3. 테이블 추출 품질

### 병합 셀(rowspan/colspan) 처리

**Azure DI** — HTML table로 출력, rowspan/colspan 완전 처리 ✅
```html
<th rowspan="2">항목</th>
<td>수치 정확</td>
```

**Docling** — Markdown pipe로 출력, 헤더 행 중복 오인식
```
| 항목 | 값 A | 값 B |
| 항목 |        ← 헤더 행 중복
| 세부항목 | 28 (4.0) | 16 (2.0) |
```

**opendataloader** — Markdown pipe, 깔끔하고 정확 ✅ (단, 스캔/한영 문서에서는 실패)

---

## 4. 처리 속도

| 파일 | Docling | Azure DI | opendataloader |
|------|---------|----------|----------------|
| 문서A (14p) | 6.0초 | 9.6초 | **2.5초** |
| 문서B (44p) | 24.6초 | **13.4초** | 3.1초 |
| 문서C (1p, 스캔) | 58.6초 ❌ | **7.0초** | 2.5초 ❌ |
| 논문D (5p) | 3.8초 | 8.5초 | 2.4초 ❌ |

> Azure DI 속도는 API 네트워크 레이턴시 포함. 문서 규모가 클수록 Docling보다 오히려 빠릅니다.

---

## 5. 종합 비교

| 평가 항목 | Docling | Azure DI | opendataloader |
|-----------|---------|----------|----------------|
| **헤더 계층 구조** | ❌ H2만 | ✅ H1~H4 | △ 문서별 편차 |
| **스캔 PDF OCR** | ❌ | ✅ | ❌ |
| **테이블 정확도** | △ 셀 오인식 | ✅ rowspan 완전 | ✅ (디지털만) |
| **한/영 혼합** | ✅ | ✅ | ❌ 실패 |
| **속도 (디지털)** | 보통 | 느림(API) | 가장 빠름 |
| **의존성** | Python ML | 클라우드 API | Java 11+ |
| **비용** | 무료 | $10/1,000p | 무료 |
| **RAG 계층 청킹** | ❌ 불가 | ✅ 바로 가능 | △ 제한적 |

---

## 6. 케이스별 권장 파서

| 케이스 | 권장 | 이유 |
|--------|------|------|
| 디지털 PDF (한국어 문서) | **Azure DI** | 헤더 계층 + 테이블 완전성 |
| 스캔 PDF | **Azure DI** | 유일하게 동작 |
| 한/영 혼합 문서 | **Azure DI 또는 Docling** | 둘 다 정상 동작 |
| 비용이 최우선 + 디지털 PDF만 | **Docling** | 무료, 단 청킹 전략 제약 |

### opendataloader를 선택하지 않은 이유

속도는 압도적(2~3초)이나 핵심 케이스에서 탈락합니다:
- 스캔 PDF 미지원
- 한/영 혼합 문서 처리 실패
- 문서별 헤더 편차가 너무 커서 RAG 청킹 전략을 일관되게 설계할 수 없음

---

## 결론

**RAG 파이프라인이라면 Azure Document Intelligence를 기본으로 선택**하는 것이 현실적입니다. 비용은 월 수십 달러 수준이지만, 헤더 계층 구조 보존 덕분에 청킹 품질이 크게 올라갑니다.

Docling은 비용 제약이 있을 때 디지털 PDF + 평탄 청킹 전략을 수용할 수 있다면 대안이 됩니다. 다만 스캔 PDF나 복잡한 테이블이 많은 문서에는 적합하지 않습니다.

```
추천 아키텍처:
디지털/스캔 PDF → Azure DI → MarkdownHeaderTextSplitter → 벡터 DB
XLSX            → openpyxl (별도 처리)
PPTX            → 추가 테스트 필요
```
