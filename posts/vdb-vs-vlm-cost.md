# RAG 시스템 비용 최적화: VDB vs VLM

RAG 파이프라인을 운영하다 보면 "이 구조가 실제로 얼마나 드는가?"를 제대로 계산해본 적이 없는 경우가 많습니다. 이 글에서는 **VDB(벡터 데이터베이스) + LLM 리랭커** 방식과 **VLM 직접 처리** 방식의 실제 월간 비용을 산정하고, 아키텍처 트레이드오프를 정리합니다.

---

## 전제 조건

| 항목 | 값 |
|------|-----|
| 총 문서 | ~225개 / 1.4GB |
| 파일 평균 페이지 | ~28장 |
| 신규 문서 (월) | 20건 |
| 일일 질의 | 100건 → **월 3,000건** |

---

## 현재 VDB 아키텍처

```
Query → Multi-Query 생성 → 벡터 검색
      → LLM Reranker (문서 1개당 LLM 1회 호출)
      → 답변 생성 LLM
```

### 멀티쿼리 + 리랭커 구조의 실제 비용

LLM이 에이전트 프롬프트 지시에 따라 `vdb_search` 툴을 병렬로 여러 번 호출합니다:

```
LLM이 vdb_search 3번 병렬 호출
  └─ 각 호출: top_k=5 검색 → 리랭커 5회 실행
총 리랭커 LLM 호출: 3 × 5 = 15회 / 사용자 질의
```

리랭커는 청크 1개당 LLM을 1회 독립 호출합니다(배치 아님):
- 입력 토큰: 프롬프트(~450) + 질의(~50) + 청크(~800) = **~1,400 tokens/호출**
- 출력: 점수 JSON (~20 tokens)

### VDB 월간 비용 (멀티쿼리 3회, top-5 기준)

| 항목 | 월 비용 |
|------|---------|
| 신규 문서 임베딩 | $0.03 |
| 텍스트 정제 LLM | $0.05 |
| **리랭커** (15 docs × 3,000 질의) | **$123** |
| 답변 생성 LLM | $42 |
| **합계** | **~$165/월** |

> 리랭커가 전체 비용의 75%를 차지합니다. 멀티쿼리 횟수 × top_k = 리랭커 호출 수이므로 선형적으로 증가합니다.

---

## 제안 VLM 아키텍처

```
전처리: 모든 파일 → VLM → 파일별 메타데이터/주석 생성
질의:  Query → 주석 임베딩 검색 → 파일 선택 → VLM 파일 전체 처리 → 답변
```

VLM(Vision-Language Model)은 PDF/PPTX를 이미지로 처리하므로 파싱 + 청킹 없이 **문서 전체를 그대로 이해**합니다.

### 파일 선택 전략 비교

VLM 방식에서 "어떤 파일을 VLM에 넘길까"를 결정하는 방법이 비용을 크게 좌우합니다.

| | VDB 리랭커 | 정규표현식 매핑 | LLM 파일 선택 |
|--|:---:|:---:|:---:|
| LLM 호출/질의 | **15회** | **0회** | **1회** |
| 월 비용 | $123 | $0 | $26 |
| 의미적 이해 | ✅ | ❌ 키워드 의존 | ✅ |

**정규표현식 방식**의 트레이드오프:
- 잘 됨: "A제품 부작용" → product=A, topic=부작용 매핑
- 취약: "이 제품이랑 다른 제품 겹치는 거 없어?" → 의도가 태그 구조 밖

### VLM 토큰 비용 계산

Gemini 2.5의 PDF/PPTX는 **258 토큰/페이지 고정** 요금이 적용됩니다(이미지 타일링 방식 아님).

```
평균 2.5개 파일 × 28페이지 = 70페이지/질의
70 × 258 tokens = 18,060 tokens/질의
3,000 질의/월 → 54.18M input tokens/월
```

### VLM 월간 비용

| 항목 | Gemini 2.5 Flash | Gemini 2.5 Pro |
|------|:----------------:|:--------------:|
| 신규 문서 주석 | $0.05 | $0.20 |
| VLM 질의 처리 (input) | $16 | $68 |
| VLM 질의 처리 (output) | $4 | $15 |
| 파일 선택 (regex) | **$0** | **$0** |
| **합계 (regex 방식)** | **~$20/월** | **~$83/월** |

## 핵심 인사이트

### VDB 방식
- **장점**: 청크 단위 처리로 파일 크기 무관, 비용 예측 가능
- **단점**: 리랭커 비용이 생각보다 훨씬 큼 (문서 수 × LLM 호출)
- **리스크**: 멀티쿼리 수가 늘면 리랭킹 대상이 급증

### VLM 방식
- **장점**: 표/이미지 포함 문서에서 정보 손실 최소, 문서 구조를 그대로 이해
- **단점**: 파일 크기 편차에 따른 비용 변동
- **리스크**: 100장 이상 대형 파일 선택 시 비용 폭발, 응답 지연

### 실제로 고려할 만한 아키텍처: 하이브리드

```
1단계: VDB로 후보 파일/청크 선별 (리랭커 없이 유사도만)
2단계: 선별된 소수 파일만 VLM 정밀 처리
→ 페이지 수 상한선 적용 필수 (예: 50p 초과 파일은 VDB 청크만 사용)
```

리랭커를 제거하고 VDB를 "파일 필터" 역할로만 사용하면, VDB의 비용 예측 가능성과 VLM의 문서 이해 능력을 동시에 활용할 수 있습니다.

---

## 결론

**LLM 기반 리랭커는 생각보다 비쌉니다.** `top_k × 멀티쿼리 수 × 질의 수`가 전부 LLM 호출로 직결되기 때문입니다.

VLM 방식은 초기엔 낯설지만, 표와 이미지가 많은 문서를 다루는 RAG에서는 파싱 손실을 줄이면서 비용도 낮출 수 있는 현실적인 대안입니다. 단, 파일 크기 상한 정책은 반드시 함께 설계해야 합니다.
